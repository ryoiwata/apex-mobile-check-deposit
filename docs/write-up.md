# Mobile Check Deposit System — Architecture Write-Up

## Architecture Choices

The system is a single Go binary with seven internal packages (`vendor/`, `funding/`, `ledger/`, `state/`, `settlement/`, `operator/`, `deposit/`), each behind an interface boundary. Go was chosen because Apex's platform is primarily Go, and the concurrency model maps well to parallel rule evaluation and batch settlement processing. PostgreSQL backs the transactional core — foreign key constraints enforce referential integrity across transfers, ledger entries, and audit logs, and ACID transactions guarantee that the critical section (state transition + ledger posting + audit log) is atomic. Redis handles duplicate detection (SHA256 hash with 90-day TTL) and rate limiting (sliding window, 10/min per account). The monolith-with-packages structure provides the same separation of concerns as microservices without the operational overhead of service discovery, network serialization, and multiple containers. Each package interface is designed for promotion to a standalone service — `vendor.NewStub()` becomes `vendor.NewHTTPClient(url)` with one line changed in `main.go`.

## Vendor Service Stub Design

The stub returns seven deterministic responses keyed by the last four characters of the account ID: `*1001` → blur, `*1002` → glare, `*1003` → MICR failure, `*1004` → duplicate, `*1005` → amount mismatch, `*1006` → clean pass, `*0000` → basic pass. This design requires zero code changes or configuration tweaks to switch scenarios — different account IDs trigger different responses automatically, and tests can run in any order with no shared state. Each response includes structured data (IQA result, MICR fields, OCR amount, confidence scores, transaction ID) so downstream services receive realistic payloads. IQA failures return actionable retake guidance; flagged items include the flag reason for operator review; clean passes populate all MICR fields for settlement file generation.

## State Machine Rationale

The transfer state machine enforces eight states (Requested → Validating → Analyzing → Approved → FundsPosted → Completed, with Rejected and Returned as terminal states) with optimistic locking: `UPDATE transfers SET status=$1 WHERE id=$2 AND status=$3`. If zero rows are affected, another process won the transition and `ErrInvalidStateTransition` is returned. The critical invariant — no deposit reaches FundsPosted without passing both vendor validation and funding service business rules — is enforced by the transition graph itself; there is no direct path from Requested or Validating to FundsPosted. The Funding Service uses a collect-all validation approach: every business rule (deposit limit, contribution cap, duplicate detection, account eligibility) is evaluated regardless of prior failures, and all violations are returned in a single response. This eliminates the frustrating fix-one-resubmit-hit-another loop. Every non-terminal error path loops back — IQA failures prompt retake, rule failures allow resubmission with all issues surfaced, operator rejections allow new deposits, and returned checks allow resubmission with a different check.

## Risks and Limitations

The vendor integration is fully stubbed — no real image processing, MICR reading, or OCR occurs. Authentication uses static bearer tokens from environment variables with no account-level access control. The system runs as a single node with no distributed locking for settlement batching. Check images and settlement files are stored unencrypted on Docker volumes. The EOD cutoff (6:30 PM CT) does not account for weekends or bank holidays. Redis graceful degradation means a duplicate check can be missed if Redis is down during the first deposit and recovers before the second. The deposit pipeline uses separate transactions for early state transitions, so a crash between `Requested → Validating` and the vendor call leaves an orphaned transfer. All data is synthetic — no real PII, routing numbers, or check images. This system has not been reviewed for Regulation CC, NACHA, PCI-DSS, or FFIEC compliance. Full risk analysis with production paths is documented in `docs/risks.md`.
